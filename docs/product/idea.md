# ゲーム用クリエイティブ生成ツール

## Why now

ゲーム制作では、実装より前後して大量のアセットが必要になる。  
UIパーツ、アイコン、背景部品、装飾、カード枠、ボタン、エフェクト、敵差分など、細かい素材が常に不足しやすい。  
個人開発や少人数開発では、この量産作業が強いボトルネックになる。

## Problem

ゲーム用アセット制作は、単に1枚絵を生成するだけでは足りない。  
同一テイストで複数サイズ、複数状態、複数属性の差分を揃える必要がある。  
汎用画像生成だけでは、統一感、透過、パーツ分離、量産性、再編集性が弱く、結局手作業が多く残りやすい。

## User

- 個人ゲーム開発者
- 少人数インディーチーム
- プロトタイプ段階で仮アセットを大量に必要とする人
- デザイナー不在、またはアセット制作工数を節約したいチーム

## Core Idea

ゲームの世界観、解像度、UIテーマ、色数、アート方向を登録しておくと、それに沿ったアセットを半自動で生成する。  
たとえば以下をまとめて扱う。

- UIボタン
- アイコン
- カード枠
- 背景全景
- 背景パーツ
- タイルセット
- エフェクト素材
- 敵やアイテムの差分素材

単なる画像生成ではなく、スタイルの統一、差分量産、透過書き出し、サイズ調整、派生バリエーション生成まで一気通貫でやるのがコア。

## MVP Focus

最初の MVP は、ゲーム内で消費頻度が高く、かつ量産と統一感の価値が出やすい 3 種に絞る。

- UIボタン / 小型パネル
- アイコン
- カード枠 / スロット枠

この 3 種なら、透過、状態差分、サイズ違い、9-slice、量産テンプレートの価値を早い段階で検証しやすい。  
逆に、複雑なタイルセット、3D、アニメーション付きエフェクトは初期対象から外す。  
背景全景はやりたい対象として残すが、MVP 本体とは別トラックで仕様化する。

詳細仕様は [spec.md](./spec.md) に分離し、生成フロー、世界観プリセット、書き出し形式、データ構造、MVP の削り方を固定する。

## Value

- ゲーム制作中のアセット不足を埋めやすい
- 仮素材から本番寄り素材まで段階的に使える
- 同一テイストの差分をまとめて作りやすい
- プロトタイプの見た目を早く整えられる
- 小規模開発でも量産が必要な素材に対応しやすい

## Requirements

- ゲームごとのビジュアル方針をプリセットとして持てる
- 透過PNGなど、ゲームで使いやすい形式で書き出せる
- 同一カテゴリの差分を量産できる
- アイコン、UI素材、背景パーツなどカテゴリ別テンプレートを持てる
- 画像生成AIを使う場合でも、最終的にサイズや余白や色味を調整しやすい
- `画面単位のKV + 素材の仕様書` だけで、別実装者が画面を再現できる
- JSON バンドルの読み込み、または仕様書と画像を置いたフォルダ指定で使える
- 初期段階では3Dモデルや複雑なアニメーションより、2D素材を優先する
- 初期段階では販促物ではなく、ゲーム内利用アセットを主対象にする
- 初期段階では UIボタン / アイコン / カード枠 の 3 種で量産ワークフローを成立させる
- 背景全景は拡張対象として持ち、MVP 後にレイヤー分離と差分運用を前提に扱う

## Risks

- 汎用デザインツールとの差別化が弱いと刺さりにくい
- AI生成だけでは細かい品質要求に応えにくい
- 一貫した画風を保てないと実用性が下がる
- パーツ分離や透過が甘いとゲームに組み込みにくい
- ゲームごとのトンマナ差が大きく、テンプレートが浅いと使われなくなる

## Open Questions

- ピクセルアート対応を最初から入れるか
- 既存素材から似たスタイルを学習・継承するか
- エンジン別の書き出しプリセットを最初から持つか
- ラスタ中心で始めた後に、どこまでベクター風 UI に寄せるか
- 背景全景を 1 枚絵として扱うか、レイヤー前提で扱うか

## Next Actions

- `spec.md` を基準に、世界観プリセット入力画面とバリエーション確認画面のワイヤーを切る
- `screen KV` と `素材仕様書` の schema を先に固定する
- AI が `critique / lock suggestion / diff regenerate suggestion` を返すループを磨く
- ボタン / アイコン / カード枠の 3 テンプレートをモック実装する
- 透過PNG、9-slice JSON、スプライトシート JSON の最小書き出しを試作する
- 背景全景の第2フェーズ仕様として、レイヤー構成と差分ルールを定義する
- 既存の仮アセット運用フローにどう差し込むか、プロトタイプ利用者へヒアリングする
